## 1. Top findings (fix immediately)

### 1.1 Seed inconsistency: `admin@start.test` is not an admin
The user I was asked to sign in as (`admin@start.test` / `test123`) exists, but has role **`volunteer`**
in the `profiles` table. The real admin from the seed migration is `admin@startcrew.test` / `StartCrew123!`.
`felix.lorenz@startglobal.org` is also a volunteer without a name. Two effects:
- Logging in with the "documented" admin credentials lands on `/volunteer`, not `/admin` â€” confusing.
- Both test users have an empty `name` field â†’ the admin user table shows `â€”`.
- **Fix:** Either promote `admin@start.test` to `admin` or remove it from the dataset.

### 1.2 Next.js middleware is never registered
`src/proxy.ts` exports `proxy()`, but Next.js only recognizes `src/middleware.ts` with `export function middleware`.
The file is silently ignored â†’ session refresh on requests never runs, and any "middleware defense" does not exist.
**Fix:** Rename the file to `src/middleware.ts` and rename the function to `middleware`.

### 1.3 Privilege escalation via `/api/auth/signup`
`src/app/api/auth/signup/route.ts:12,43-52` takes `role` from the request body and forwards it as `user_metadata.role`.
If the DB trigger `handle_new_user()` writes that field into `profiles.role`, any anonymous caller can
`POST {"role":"admin"}` and create an admin account.
**Fix:** Force role server-side to `"volunteer"` or restrict the route to authenticated admins.

### 1.4 Team-Lead view: `profiles` query is not team-scoped
`src/app/(dashboard)/lead/page.tsx:142-146` filters the roster `profiles` query only with `.eq("is_active", true)`.
Every lead therefore receives `id/name/role/avatar` of **every active user** in the Server-Component payload â€”
including other teams and admins. Violates the CLAUDE.md rule
"Every Supabase table query filters by role/team EXPLICITLY".
**Fix:** `.in("id", volunteerIdsFromTeamAssignments)` or similar.

### 1.5 Volunteer view: all phone numbers end up in the payload
`src/app/(dashboard)/volunteer/page.tsx:90-91` pulls `profiles(phone)` unfiltered.
The UI respects the rule "lead phone only after commit", but the raw payload contains every active phone number.
Privacy leak.
**Fix:** Restrict to `role='lead'` of the relevant zone.
